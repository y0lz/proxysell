Техническое задание (ТЗ): Telegram-бот агрегатор и чекер прокси (Anti-Block Bot)
1. Описание проекта
Создание Telegram-бота, который автоматически собирает (скрапит) открытые MTProto и SOCKS5 прокси из GitHub-репозиториев и интернета, проверяет их на работоспособность (чекер) и выдает конечным пользователям в формате удобных deep-links (ссылок для автоматического применения в клиенте Telegram).
Бизнес-модель: Freemium (1 бесплатный прокси в сутки) + микро-подписка за доступ к пулу самых быстрых VIP-прокси и кнопке мгновенной замены заблокированного адреса.

2. Технологический стек
Язык программирования: TypeScript (Node.js).

Фреймворк для бота: grammY — обеспечивает полную поддержку статической типизации и самых свежих обновлений Telegram Bot API 9.6.

База данных (MVP): SQLite (локальный файл).

База данных (Масштабирование): Supabase (PostgreSQL).

ORM: Prisma или Drizzle ORM (критически важно использовать ORM со старта, чтобы при миграции с SQLite на Supabase потребовалось лишь изменить строку подключения в .env и не переписывать SQL-запросы).

Скрапинг и сеть: axios (с использованием socks-proxy-agent для внутренних проверок).

3. Архитектура системы (Микросервисный подход внутри монолита)
Проект должен быть разделен на три независимых модуля, работающих параллельно:

Модуль 1: Скрапер (Scraper)
Запускается по расписанию (Cron) каждые 2–3 часа. Его задача — скачивать списки прокси из открытых источников и сохранять новые уникальные IP в БД со статусом unchecked.

Источники MTProto:

Парсинг raw-файлов с GitHub: репозиторий SoliSpirit/mtproto (обновляется каждые 12 часов).

Форки-зеркала: Grim1313/mtproto-for-telegram.

Источники SOCKS5:

Репозитории-агрегаторы: proxifly/free-proxy-list (обновляется регулярно).

Опционально можно интегрировать логику Python-скриптов вроде NotAnyoneMe/ProxyChecker для адаптации под Node.js.

Модуль 2: Чекер (Validator)
Постоянно работающий фоновый процесс (Worker), который берет прокси из базы данных и пытается установить через них соединение с серверами Telegram (api.telegram.org).

Метрики: Успешность соединения и замер задержки (Latency/Ping).

Статусы в БД: active, dead, slow.

Очистка: Если прокси не отвечает 3 проверки подряд, он удаляется из SQLite, чтобы не раздувать базу.

Модуль 3: Telegram-бот (grammY)
Фронтенд для пользователя.

Интерфейс: Использование актуальных фич API. Например, кастомные эмодзи на кнопках интерфейса (icon_custom_emoji_id из API 9.4) для красивого визуального оформления кнопок выбора протокола.

Генерация ссылок:

Для MTProto бот формирует строку: tg://proxy?server=<IP>&port=<PORT>&secret=<SECRET>.

Для SOCKS5 бот формирует строку: tg://socks?server=<IP>&port=<PORT>&user=<USER>&pass=<PASS>.

4. Пользовательский путь (UX/UI)
/start: Бот приветствует пользователя, коротко объясняет разницу (MTProto сложнее заблокировать через DPI, SOCKS5 работает для всего трафика).

Главное меню (Inline-клавиатура):

🟩 Получить бесплатный прокси (MTProto / SOCKS5 на выбор).

🚀 Купить VIP доступ (Безлимитные супер-быстрые прокси).

❓ Помощь / Как настроить.

Логика бесплатной выдачи: Бот проверяет ID пользователя в SQLite. Если сегодня он уже брал бесплатный прокси — выдается отказ с предложением купить VIP.

Сценарий VIP-клиента: Нажатие кнопки «Заменить прокси» мгновенно достает из БД прокси со статусом active и пингом менее 100мс (самые сливки, собранные чекером).

5. Платежная инфраструктура
Для приема платежей используем Telegram Payments API.

В зависимости от ваших возможностей, подключается фиатный шлюз (например, YooMoney или Stripe для международных платежей) или криптовалютный (TON Connect / CryptoPay).

Важно: Использование Telegram Stars здесь не рекомендуется из-за 21-дневного холда на вывод средств , так как вам могут понадобиться деньги на оплату серверов в начале каждого месяца.

6. План масштабирования
Фаза 1 (Месяц 1): MVP. Запуск всего стека на одном дешевом сервере Hetzner (за ~$5/мес). Бэкенд на Node.js, локальный файл database.sqlite, Telegram Webhooks.

Фаза 2 (Месяц 2): Маркетинг и первые пользователи. Распространение ссылок на бота в профильных чатах и сабреддитах, где пользователи жалуются на блокировки и неработающие VPN.

Фаза 3 (Рост > 1000 пользователей): Файл SQLite начнет блокироваться при частых одновременных записях от чекера и чтениях от бота. На этом этапе в файле конфигурации ORM база SQLite заменяется на облачную PostgreSQL от Supabase (Free Tier вполне хватит для начала). Код бота при этом остается неизменным.

7. Риски и их митигация
Риск блокировки IP вашего сервера от РКН: Поскольку бот сам постоянно "стучится" к серверам Telegram для проверки прокси, его трафик может быть замечен.

Решение: Хостить скрипт бота и чекера строго на зарубежном сервере, который не подвержен локальным фильтрам DPI.

Пустые базы: Бесплатные прокси в интернете быстро "умирают".

Решение: Чекер должен работать асинхронно и агрессивно в многопоточном режиме (через Promise.allSettled в Node.js), пингуя сразу сотни IP, чтобы всегда держать в базе хотя бы 20-30 рабочих прокси для выдачи клиентам.
